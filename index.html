技術專題：AWS 雲端環境安全與 Python Flask 全鏈路部署實作

📌 一、 專案目錄結構調查報告：src 與 test

在現代軟體工程中，良好的目錄結構能大幅提升協作效率。本專案核心劃分為兩個最關鍵的資料夾：

1. src 資料夾 (Source Code / 原始碼)

  • 定義： 專案的核心原始碼資料夾。
  • 主要功能： 存放維持應用程式核心運作的正式程式碼（如 Flask 後端邏輯、HTML/CSS 畫面、核心演算法）。
  • 重要性： 這是系統成品的本體，在進行編譯、打包與雲端上線部署時，系統主要只會讀取此處。

2. test 資料夾 (Test Code / 測試碼)

  • 定義： 自動化測試資料夾（通常與 src 結構對稱）。
  • 主要功能： 存放用來檢驗原始碼是否正確執行的自動化測試腳本（如單元測試、整合測試）。
  • 重要性： 專案的品質把關者。測試碼不會被部署給使用者看見，但能確保開發人員在修改核心程式碼時「不產生舊功能的 Bug」。

🔒 二、 雲端安全防護防線：AWS 根帳號與 Linux 伺服器強化

雲端運算（如 AWS）雖然便利，但若 Root 帳號流失將造成毀滅性損失。本實作歸納出兩大環境的安全防護重點：

情境一：AWS 雲端根帳號 (Root User) 強化

  • 啟用多重要素驗證 (MFA)： 透過 Google Authenticator 等虛擬通行密鑰綁定，建立密碼外洩後的第二道安全防線。
  • 刪除 Root 存取金鑰 (Access Key)： 絕不建立也不外流 access_key_id，防止惡意駭客搬空雲端資源。
  • 建立 IAM 使用者群組： 實務上將 Root 帳號封印，改建立具備管理權限的 IAM 使用者（如建置 admin、DevOps 權限組）進行維運。
  • 預算告警控制： 於 Billing 建立「零預算金額」或「每月成本預算（如設定 3 美金，約 100 元台幣）」，一旦花費達 80% 即發送電郵通知，防止產生龐大費用。

情境二：Linux 伺服器 Root 帳號防護

  • 禁止 Root 遠端登入： 編輯 /etc/ssh/sshd_config 將 PermitRootLogin 改為 no，防止自動化機器人暴力破解密碼。
  • 改用 SSH 金鑰憑證： 關閉密碼登入功能（PasswordAuthentication no），僅允許配對金鑰對（Key Pair）登入。

☁️ 三、 從實體到虛擬：AWS EC2 核心雲端元件對照

實作中利用 AWS 東京機房進行雲端佈署，其底層架構包含以下核心元件之中英對照：

AWS 雲端元件 中文功能對照 實務技術職責
Instance EC2 虛擬伺服器實例 負責專案的主要運算與應用程式運作流程。
EBS 雲端區塊儲存體/虛擬硬碟 伺服器的硬碟，負責持久化存放系統檔案與核心資料。
AMI 系統映像檔/樣板 包含作業系統的開機懶人包（例如部署 Windows Server 2025）。
Security Group 安全群組/虛擬防火牆 控制通訊埠（如允許 RDP、HTTPS 流量）進出的第一道閘口。
Key Pair 金鑰對/密鑰 遠端安全登入 Windows 或 Linux 伺服器的必備電子鑰匙（如 .pem 檔案）。

🐍 四、 後端開發實戰：Python Flask Web 應用部署

本專案基於 Python 3.12 進行開發，在遠端 Windows 伺服器成功架設了 Flask 服務狀態儀表板 (Flask App Console)，運行於 Port 19191。

部屬與執行流程：

# 1. 透過 Git 將雲端倉庫複製到本機環境
git clone https://github.com/figfhgfgg/my-ai-project.git

# 2. 移動到專案資料夾並利用環境配置下載套件
cd my-ai-project
pip install -r requirements.txt

# 3. 啟動後端 Web 應用服務
python run.py

成果：成功開啟一鍵瞬開瞬關的容器化部署與高併發架構環境，儀表板可即時監控伺服器 Online 狀態與 Python 環境資訊。

🌐 五、 延伸資訊觀念：URL 與 URN 的差別

在網路資源識別（URI）家族中，URL 與 URN 的核心差別如下表：

特性 URL (統一資源定位符) URN (統一資源名稱)
主要功能 告訴系統資源「在哪裡」 (Where) 告訴系統資源「是什麼」 (What)
持久性 較低。搬移檔案或變更網域就會失效 (404) 極高。名稱是全宇宙永久且獨一無二的。
